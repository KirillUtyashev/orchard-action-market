# Value Comparison Tool — Design

## Overview

A debugging/analysis tool that loads two trained checkpoints (possibly different learning types, encoder types, architectures), generates a shared set of states, and compares their team-value predictions state-by-state. Produces an interactive HTML report.

Entry point: `python -m orchard.compare_values`

---

## Input: Two Run Directories

Each run directory (produced by `train.py`) contains:
```
runs/2025-07-01_143000/
  metadata.yaml       # full config + run info
  checkpoints/
    final.pt           # or step_N.pt
  metrics.csv
```

The tool takes two such directories (called **run A** and **run B**) and:
1. Loads `metadata.yaml` from each → extracts `ExperimentConfig`.
2. **Validates env compatibility**: the env configs must match exactly (height, width, n_agents, n_apples, gamma, r_picker, force_pick, max_apples, env_type, stochastic params). These determine the state space and reward structure — if they differ, value comparison is meaningless.
3. Things that **may differ**: model architecture, encoder type, learning type (centralized vs decentralized), train hyperparameters, td_target, train_method, etc.

### Why run directories, not raw config + checkpoint?

Each run dir bundles the config that was *actually used* for training (in metadata.yaml) together with the checkpoint. This prevents mismatches between a config file and a checkpoint trained under a different config. The viz tool already has `_load_config_or_metadata` for this pattern.

## CLI

```
python -m orchard.compare_values RUN_DIR_A RUN_DIR_B [options]
```

### Required
- `RUN_DIR_A` — path to first run directory
- `RUN_DIR_B` — path to second run directory

### Optional
- `--checkpoint-a NAME` — checkpoint filename within RUN_DIR_A/checkpoints/ (default: `final.pt`)
- `--checkpoint-b NAME` — checkpoint filename within RUN_DIR_B/checkpoints/ (default: `final.pt`)
- `--n-states N` — number of states to generate for comparison (default: 200)
- `--rollout-steps N` — steps of nearest-apple rollout for state generation (default: n_states * 20)
- `--seed S` — seed for state generation (default: 42)
- `--output-dir DIR` — where to write output (default: `./compare_output/`)
- `--dpi N` — PNG render DPI (default: 120)

### Examples
```bash
# Basic: compare two runs
python -m orchard.compare_values runs/decentralized_001 runs/centralized_001

# Compare specific checkpoints (not final)
python -m orchard.compare_values runs/exp1 runs/exp2 --checkpoint-a step_50000.pt --checkpoint-b step_50000.pt

# More states, custom output
python -m orchard.compare_values runs/exp1 runs/exp2 --n-states 500 --output-dir ./analysis/
```

## State Generation

States are generated by running nearest-apple policy from a seeded initial state, collecting unique **after-states** (since both after-state and pre-action models are predicting V of states encountered during play).

We reuse `collect_after_state_test_states` from `eval.py` for after-state TD target runs. For pre-action TD target, we use `collect_on_policy_test_states`.

**Which td_target governs state collection?** Since the two runs may use different td_targets, we need a convention. The td_target determines *what kind of state* the model's V̂ is defined over:
- `after_state`: V̂(s) where s is a post-action, pre-spawn state.
- `pre_action`: V̂(s) where s is a pre-action state (actor about to move).

If both runs use the same td_target, no issue — collect that kind of state.

If they differ, the models are literally predicting values of different state types, so direct comparison of V̂_A(s) vs V̂_B(s) on the same state `s` is still meaningful as long as we're clear about what `s` is. We collect **both** types and compare each model on the state type it was trained on... but that makes cross-comparison harder.

**Simplification for v1:** Require both runs to use the same td_target, or default to after-state collection and warn if there's a mismatch. In practice, all your current experiments use `after_state`.

## Value Computation

### Team Value

For a given state `s`:

**Decentralized** (N networks, each with its own encoder):
```
V_team(s) = Σ_{i=0}^{N-1} V_i(s)
```
where `V_i(s)` encodes from agent i's perspective and evaluates with network i.

**Centralized** (1 network, centralized encoder):
```
V_team(s) = V_0(s)
```
The single network directly predicts team value.

### Encoder Handling

Critical: the two runs may use different encoder types. The global `encoding.init_encoder` singleton can't serve both simultaneously.

**Solution:** Use `encoding._create_encoder(encoder_type, env_cfg, k)` to create two separate encoder instances. Each model's forward pass uses its own encoder:

```python
encoder_a = encoding._create_encoder(cfg_a.model.input_type, cfg_a.env, cfg_a.model.k_nearest)
encoder_b = encoding._create_encoder(cfg_b.model.input_type, cfg_b.env, cfg_b.model.k_nearest)

# For model A (decentralized):
v_team_a = sum(net_a[i](encoder_a.encode(s, i)) for i in range(n_agents))

# For model B (centralized):
v_team_b = net_b[0](encoder_b.encode(s, 0))
```

We should also expose `_create_encoder` as a public function (rename to `create_encoder`) since this tool has a legitimate need for multiple encoder instances.

## Per-State Record

For each state, we compute and store:

```python
@dataclass
class StateComparison:
    state: State
    state_index: int

    # Model A
    team_value_a: float
    agent_values_a: dict[int, float]   # {agent_idx: V_i(s)} — N entries if decentralized, 1 if centralized
    label_a: str                        # e.g. "decentralized (run 2025-07-01_143000)"

    # Model B
    team_value_b: float
    agent_values_b: dict[int, float]
    label_b: str

    # Derived
    team_diff: float                    # V_team_a - V_team_b
    team_abs_diff: float                # |V_team_a - V_team_b|
```

## HTML Output

A single self-contained HTML file (like viz) with:

### 1. Summary Header
- Run A and Run B labels (directory names, learning type, encoder, training steps)
- Number of states compared
- Aggregate stats: mean |diff|, max |diff|, mean diff (signed bias), correlation

### 2. Scatter Plot: V_team_A vs V_team_B
- Each dot is one state
- Diagonal line = perfect agreement
- Color by |diff| (red = large disagreement)
- Hover shows state index

### 3. Difference Histogram
- Distribution of (V_team_A - V_team_B)
- Shows systematic bias direction

### 4. Difference by State Plot
- Bar chart or line: states sorted by team_diff
- Quick visual for where the models disagree most

### 5. Per-Agent Scatter (if both decentralized)
- One scatter per agent: V_i_A vs V_i_B
- Shows whether disagreement is driven by specific agents

### 6. State Browser (sortable table)
- Columns: state_index, V_team_A, V_team_B, diff, |diff|, per-agent values
- Sortable by any column (click headers)
- Top 10 worst-agreement and top 10 best-agreement highlighted
- Clicking a row shows the rendered grid for that state

### 7. Grid Renderings for Extreme States
- Top K states with largest |diff| rendered as PNGs (embedded)
- Top K states with smallest |diff| rendered
- Shows what the actual board looks like when models agree/disagree

## Env Compatibility Check

Fields that must be identical between run A and run B:
- `height`, `width`, `n_agents`, `n_apples`, `gamma`
- `r_picker`, `force_pick`, `max_apples`
- `env_type` and full `stochastic` config (if stochastic)

The check should produce a clear error message listing which fields differ.

## Possible Future Extensions

- **MC ground truth column**: Add `--ground-truth` flag that runs MC rollouts under nearest-apple. Works cleanly for value-learning mode under nearest-apple policy. Would add a third column to every comparison.
- **Multiple checkpoints from same run**: Compare step_10000 vs step_50000 to see learning progression.
- **State loading from file**: Save/load state lists for reproducibility across runs.
- **Per-cell value heatmap**: For small grids (2x2, 3x3), render a heatmap of V_team over all possible agent positions with apples fixed.
